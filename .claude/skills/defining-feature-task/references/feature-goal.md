# Feature 目標の定義

## 目標の粒度: 振る舞いレベル

Feature の目標は **機能としての振る舞い** で定義する。
「何が実現可能になるか」を、実装の詳細ではなくシステムの振る舞いとして記述する。

---

## 目標の種類と定義方法

### バックエンド Feature

GraphQL レベルでの input/output を振る舞いの境界とする。

**良い目標の例:**
- `createTrial` mutation に `steps` フィールドを含めて送信すると、Trial が Step 付きで作成される
- `trial` query で `steps { position, parameters }` が取得できる
- 存在しない project_id を指定した場合、適切なエラーレスポンスが返る

**検証方法:** GraphQL 統合テスト（`tests/graphql/`）のパス

### フロントエンド Feature

UI の変化やバックエンドへのリクエスト変更を振る舞いの境界とする。

**良い目標の例:**
- Trial 作成画面で Step を追加・削除できる
- Step のパラメーターを入力すると、`createTrial` mutation の `steps` に含まれて送信される
- モバイル表示で入力フォームが画面幅に収まる

**検証方法:** UI の動作確認、リクエスト内容の確認

### リファクタリング Feature

既存の振る舞いが変わらないことを目標とする。

**良い目標の例:**
- リファクタリング前後で全ての既存テストがパスする
- GraphQL の input/output に変更がない
- レスポンスの構造・値が同一である

**検証方法:** 既存テストスイートの全パス

---

## 記述ルール

1. **振る舞いで書く**: 「〜ができるようになる」「〜が返る」の形式
2. **実装に言及しない**: 「Repository に〜メソッドを追加」ではなく「〜が永続化される」
3. **検証可能にする**: テストやコマンドで確認できる形にする
4. **正常系と異常系の両方を含める**: ハッピーパスだけでなくエラーケースも定義する

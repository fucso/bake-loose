## 知識の構造化パターン

知識を効果的に伝えるための構造化パターン。

### テーブル形式

比較・分類が必要な情報に最適。

| パターン | 用途 |
|---------|------|
| 判断基準テーブル | 条件に応じた選択肢の提示 |
| 比較テーブル | 複数の選択肢の長所・短所 |
| チェックリスト | 網羅的な検証項目 |

### 記載すべきこと / しないこと

ルールの境界を明確にする。

```
## 記載すべきこと
- Claude が知らないプロジェクト固有の情報
- 判断基準とデフォルト値

## 記載しないこと
- Claude が一般知識として持つ情報
- 時間依存の情報（バージョン番号など）
```

### 良い例 / 悪い例

抽象的なルールを具体的に示す。

```
**良い例:** `name: processing-pdfs` — 動名詞形、目的が明確、小文字・ハイフン区切り
**悪い例:** `name: MySkill_v2` — 大文字・アンダースコア・バージョン番号
```

## 適切な自由度の設計

タスクの脆弱性（間違えた場合のリスク）に応じて自由度を設定する。

| 自由度 | 適用場面 | 手法 |
|--------|---------|------|
| **高** (創造的) | ドキュメント執筆、アイデア生成 | ガイドラインのみ提示し、形式は自由 |
| **中** (構造化) | 仕様記述、レビュー | テンプレート + カスタマイズ可能なセクション |
| **低** (厳密) | 設定ファイル生成、検証処理 | 厳密なテンプレート + バリデーション |

### 判断基準

- **間違えた場合の影響が大きい** → 低い自由度（テンプレート + バリデーション）
- **正解が複数ある** → 高い自由度（ガイドラインのみ）
- **一貫性が重要** → 中程度の自由度（テンプレート + カスタマイズ）

## 入出力例の活用

### 例を含めるべき場面

- 出力のスタイルや形式が重要な場合
- ルールだけでは意図が伝わりにくい場合
- 「良い」と「悪い」の境界が曖昧な場合

### 例の設計原則

- **最小限**: ルールを説明するのに必要十分な長さ
- **対比**: 良い例と悪い例をセットで提示
- **理由付き**: なぜ良い/悪いかを簡潔に説明

```
**良い例:** (理由) 具体的な内容
**悪い例:** (理由) 具体的な内容
```

### 例を含めないべき場面

- Claude の一般知識で十分判断できる場合
- ルールが明確で解釈の余地がない場合

## テンプレート活用

### テンプレートを作成すべき場面

- 出力の形式が一定で繰り返し使用される
- 必須セクションが定義されている
- 一貫性が品質に直結する

### テンプレートの設計原則

- **穴埋め式**: `{{placeholder}}` で可変部分を明示
- **コメント付き**: 各セクションの記載意図を簡潔に説明
- **最小限**: 必須項目のみ含め、オプションは注記で示す

### テンプレートを作成すべきでない場面

- 出力が毎回大きく異なる（創造的なタスク）
- テンプレートが制約になり品質を下げる場合

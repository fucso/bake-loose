# アンチパターン集

スキル作成時に避けるべきパターンと改善方法。

---

| アンチパターン | 問題点 | 改善方法 |
|--------------|--------|---------|
| SKILL.md に詳細を記載 | コンテキスト圧迫。毎回全文が読み込まれる | references/ に分離し SKILL.md は ~100行に |
| 一般知識の繰り返し | トークン浪費。Claude は既に知っている | Claude が知らないプロジェクト固有の情報のみ記載 |
| Windows 形式のパス (`\`) | macOS/Linux で動作しない | `/` のみ使用 |
| 選択肢の列挙のみ | 判断基準がなく Claude が迷う | デフォルト値 + エスケープハッチ（条件付きの代替）を提示 |
| 参照の入れ子 | 読解コスト増大。どこに情報があるか追跡困難 | SKILL.md → references/ の1階層のみ |
| 曖昧な description | スキルのマッチング精度が低下 | 具体的キーワード + 「使用タイミング:」で条件を明示 |
| バージョン番号の記載 | 陳腐化する。メンテナンスコスト増 | 実行時に最新情報を取得する方法に委譲 |
| ワークフロー主体の未記載 | 誰が実行するか不明で停滞する | 各ステップに 👤/🤖/🗣️ を明示 |
| 全ステップが Claude 自動 | ユーザーの確認なしに進行しリスク | 重要な判断ポイントに 🗣️ ディスカッションを配置 |
| 巨大な単一ファイル | 読み込みコスト大。部分参照不可 | トピックごとにファイル分割 |
| テンプレートの過剰な制約 | 創造的タスクの品質が低下 | タスクの脆弱性に応じた自由度設計 |
| 入出力例の欠如 | スタイル依存の出力で品質がブレる | 良い例/悪い例をセットで提示 |

---

## 発見時の対応

アンチパターンを発見した場合:

1. 該当箇所を特定する
2. 上記テーブルの「改善方法」に従い修正する
3. `validate-skill-structure.sh` を再実行して構造を検証する

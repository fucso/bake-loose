## ワークフロー各ステップの主体明示

すべてのステップに主体（👤 ユーザー / 🤖 Claude 自動 / 🗣️ ディスカッション）を明記する。

**良い例:**

| Step | 主体 | 内容 |
|------|------|------|
| 1. 要件確認 | 🗣️ ディスカッション | 対象と範囲を確認 |
| 2. 実行 | 🤖 Claude 自動 | スクリプトを実行 |
| 3. レビュー | 🗣️ ディスカッション | 結果をユーザーと確認 |

**悪い例:**
- ステップの羅列のみで、誰が実行するか不明
- 全ステップが「Claude 自動」 — ユーザーの確認ポイントがない

## scripts/ への定型処理切り出し

### 切り出すべき処理

- ファイルの存在・形式チェック
- 命名規則の検証（正規表現）
- ファイル行数のカウント
- テンプレートからのファイル生成
- 繰り返し実行される検証処理

### 切り出すべきでない処理

- コンテキスト依存の判断（コード品質、設計妥当性）
- ユーザーの入力を必要とする対話的処理
- 1回限りの処理

### スクリプト設計の原則

- 引数でターゲットパスを受け取る（ハードコードしない）
- 成功/失敗を終了コードで返す
- エラーメッセージは具体的に（何が期待され、何が実際だったか）
- `set -euo pipefail` をスクリプト冒頭に記載

## フィードバックループの設計

ワークフロー内にフィードバックループを組み込み、品質を段階的に向上させる。

```
実行 → 検証 → [問題あり] → 修正 → 再検証
                [問題なし] → 次のステップへ
```

**設計ポイント:**
- 自動検証（scripts/）と Claude レビューを組み合わせる
- 自動検証で検出可能な問題は scripts/ に委譲する
- 判断が必要な問題は Claude レビューで対応する
- ループの最大回数を暗黙的に想定する（無限ループを避ける）

## チェックリストの活用

品質チェックを構造化するためにチェックリストを使用する。

**効果的なチェックリスト:**
- 検証項目が具体的で曖昧さがない
- 自動検証可能な項目と手動レビュー項目を分離
- 重要度順に並べる

**チェックリストの配置:**
- 自動検証項目 → scripts/ に実装
- 手動レビュー項目 → references/ にチェックリストとして配置

## 条件分岐パターン

ワークフロー内の条件分岐は、判断基準を明確にテーブルで示す。

**良い例:**

| 条件 | アクション |
|------|----------|
| テストが全件パス | Step 5 へ進む |
| テスト失敗あり | エラーを修正して再実行 |
| 環境エラー | ユーザーに報告して中断 |

**悪い例:**
- 「適切に判断する」のような曖昧な分岐条件
- デフォルトケースがない分岐

## 1. 簡潔性の原則

スキルには「Claude に知識があるかどうか」ではなく「プロジェクト内で統一・参照する価値があるか」で記載を判断する。

**記載すべきもの:**
- **プロジェクト固有の情報** — コーディング規約、アーキテクチャ決定など
- **判断基準** — 何をすべきか/すべきでないかの境界
- **デフォルト値** — 明示しないと Claude が異なる選択をする可能性がある項目
- **統一したいベストプラクティス** — 言語・フレームワーク・デザインパターンの適用方針をプロジェクト内で揃えたい場合（例: レビュー時の基準として参照、実装スタイルの統一）
- **ツールの利用方針** — ある目的に対してツールの使い方を統一したい場合（例: テストの実行方法、デプロイ手順）

**記載しないもの:**
- 上記のいずれにも該当しない、単なる一般知識の繰り返し（例: 「関数は短く保つべき」のような普遍的な原則を説明なしに列挙）

## 2. 段階的開示

SKILL.md はエントリポイントとして概要と参照リンクのみを提供する。

```
SKILL.md (概要 + リンク)
  ├── references/workflow.md (詳細手順)
  ├── references/best-practices.md (ベストプラクティス)
  └── templates/output.md (出力テンプレート)
```

Claude は必要に応じて references/ を読み込む。すべてを SKILL.md に詰め込まない。

---

## 3. SKILL.md は ~100行

SKILL.md に含めるもの:
- YAML フロントマター (name, description)
- スキルの概要（1-2段落）
- 種別や判断基準のテーブル（必要な場合）
- 参照リンク一覧

SKILL.md に含めないもの:
- 詳細なワークフロー手順
- ベストプラクティスの詳細
- 長いコード例

---

## 4. 1階層参照

references/ 内のドキュメントから別の references/ ドキュメントを参照しない。

```
✅ SKILL.md → references/workflow.md
✅ SKILL.md → references/best-practices.md
❌ references/workflow.md → references/best-practices.md
```

各 references/ ドキュメントは自己完結させる。必要な情報は該当ファイル内に記載する。

---

## 5. 100行超のファイルには目次

ファイルが100行を超える場合、冒頭に目次を配置する。

```markdown
## 目次

1. [セクション1](#セクション1)
2. [セクション2](#セクション2)
3. [セクション3](#セクション3)
```

## 6. 一貫した用語

スキル内で同じ概念に同じ用語を使用する。

**実践方法:**
- スキル内で使用する主要用語を SKILL.md の概要で定義
- references/ 全体で同じ用語を使用
- 略語を使う場合は初出時に正式名称を併記

## 7. 命名規則

スキル名 (`name` フィールド) とディレクトリ名は一致させる。

### 制約

- 小文字・数字・ハイフンのみ
- 最大64文字
- 予約語 (`anthropic`, `claude`) を含めない

### 推奨: 動名詞形 (verb + -ing)

スキルが提供する機能を動名詞形で表現する。

| 形式 | 例 | 推奨度 |
|------|-----|--------|
| **動名詞形** | `creating-agent-skill`, `processing-pdfs`, `testing-code` | 推奨 |
| 名詞句 | `pdf-processing`, `code-analysis` | 許容 |
| アクション指向 | `process-pdfs`, `analyze-code` | 許容 |

### 避けるべき命名

| パターン | 例 | 問題 |
|---------|-----|------|
| 曖昧 | `helper`, `utils`, `tools` | 何をするか不明 |
| 過度に一般的 | `documents`, `data`, `files` | マッチング精度が低い |
| 大文字・アンダースコア | `MySkill`, `my_skill` | 命名規則違反 |
| バージョン番号 | `skill-v2` | 陳腐化する |

## 8. 評価と改善

スキルの品質は反復的に改善する。

**Claude A/B パターン:**
1. スキルを適用してタスクを実行 (A)
2. 結果を評価し改善点を特定
3. スキルを修正して再度タスクを実行 (B)
4. A と B を比較して改善を確認

**改善のトリガー:**
- スキルを使用した結果が期待と異なる場合
- 同じ修正を複数回指摘される場合
- 新しいユースケースが発見された場合

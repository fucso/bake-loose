## ワークフロー概要

| Step | 主体 | 内容 |
|------|------|------|
| 1. 要件ヒアリング | 🗣️ ディスカッション | 目的・トリガー条件・対象ユーザー・スキル種別を確認 |
| 2. 設計 | 🗣️ ディスカッション | コマンド型: ワークフロー設計 / 知識型: 知識構造設計 |
| 3. 構成設計 | 🤖 Claude 自動 | 必要ファイル一覧と各ファイルの記載トピックを決定 |
| 4. **構成レビュー** | 🗣️ ディスカッション | ファイル構成と各ファイルの記載トピックをユーザーがレビュー |
| 5. SKILL.md 作成 | 🤖 Claude 自動 | テンプレートから生成。概要 + 参照リンクのみ (~100行) |
| 6. サブドキュメント作成 | 🤖 Claude 自動 | references/ の各ドキュメントを作成 |
| 7. テンプレート作成 | 🤖 Claude 自動 | templates/ のフォーマットテンプレートを作成 (必要な場合のみ) |
| 8. スクリプト作成 | 🤖 Claude 自動 | scripts/ の定型処理スクリプトを作成 (コマンド型で必要な場合のみ) |
| 9. 構造検証 | 🤖 Claude 自動 | `validate-skill-structure.sh` を実行 + Claude による内容レビュー |
| 10. 最終レビュー | 🗣️ ディスカッション | 生成物の確認・修正 |

## 各ステップの詳細

### Step 1: 要件ヒアリング 🗣️

確認すべき項目:
- **目的**: このスキルで何を達成したいか
- **トリガー条件**: どのような状況で使用されるか（description に記載する内容）
- **対象ユーザー**: Claude 自身 / 人間の開発者 / 両方
- **スキル種別**: コマンドワークフロー型 or 知識・ルール提供型

### Step 2: 設計 🗣️

→ [種別ごとの設計観点](#step-2-種別ごとの設計観点) を参照

### Step 3: 構成設計 🤖

以下を出力する:
- ディレクトリツリー（全ファイルパス）
- 各ファイルに記載するトピックの箇条書き
- 各ファイルの想定行数

### Step 4: 構成レビュー 🗣️

ユーザーに以下を確認:
- ファイル構成の過不足
- 各ファイルの記載トピックの妥当性
- 不要なファイルがないか

### Steps 5-8: ファイル作成 🤖

テンプレート (`templates/skill-md.md`, `templates/reference-doc.md`) を使用して作成。

### Step 9: 構造検証 🤖

1. `validate-skill-structure.sh` を実行して構造を自動検証
2. Claude が `quality-checklist.md` の Phase B に基づき内容をレビュー

### Step 10: 最終レビュー 🗣️

生成物をユーザーに提示し、修正があれば対応。

## Step 2: 種別ごとの設計観点

### コマンドワークフロー型

- ワークフローのステップ設計（何を・どの順で・誰が）
- フィードバックループの位置（実行 → 検証 → 修正）
- scripts/ で自動化する処理の洗い出し
- ユーザーとの対話ポイントの特定

### 知識・ルール提供型

- 知識の分類と構造化方針
- 適切な自由度レベルの決定（高/中/低）
- テンプレートの必要性判断
- 入出力例の必要性判断

## ディレクトリ構成の設計指針

| ディレクトリ | 用途 | 配置基準 |
|-------------|------|---------|
| `references/` | 詳細な知識・ルール・ワークフロー | SKILL.md に収まらない詳細情報 |
| `templates/` | 穴埋め式フォーマット | 出力の形式が決まっている場合 |
| `scripts/` | 定型処理の自動化 | 繰り返し実行される検証・変換処理 |

---

## ワークフロー主体の設計ガイド

コマンドワークフロー型では、各ステップに以下の主体を明示する:

| 主体 | 記号 | 使い分け |
|------|------|---------|
| ユーザー | 👤 | ユーザーの入力や判断が必須のステップ |
| Claude 自動 | 🤖 | Claude が自律的に実行できるステップ |
| ディスカッション | 🗣️ | ユーザーと Claude が対話して決定するステップ |

**原則**: 判断が必要なステップはディスカッション、定型処理は Claude 自動。

---

## scripts/ への切り出し判断基準

以下の **すべて** に該当する処理を scripts/ に切り出す:

1. **繰り返し実行される** — 1回限りの処理ではない
2. **定型的である** — 入力に応じた判断が不要
3. **シェルコマンドで表現できる** — 複雑なロジックが不要
4. **検証可能な出力がある** — 成功/失敗が明確

**例**: ファイル存在チェック、命名規則検証、行数カウント → scripts/ に切り出す
**反例**: コード品質の判断、設計の妥当性評価 → Claude によるレビュー

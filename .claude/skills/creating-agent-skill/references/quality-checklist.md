# 品質チェックリスト

スキル作成プロセスの検証用チェックリスト。2フェーズで構成。

※ これはスキル作成時の検証用。出来上がった知識系スキル自体には品質チェックフェーズは含まれない。

---

## Phase A: 構造チェック (自動検証)

`scripts/validate-skill-structure.sh` で自動検証される項目。

| # | チェック項目 | 検証方法 |
|---|------------|---------|
| A1 | SKILL.md が存在する | ファイル存在チェック |
| A2 | YAML frontmatter に `name` が存在 | grep パターンマッチ |
| A3 | YAML frontmatter に `description` が存在 | grep パターンマッチ |
| A4 | `name` が命名規則準拠 | 正規表現: `^[a-z0-9][a-z0-9-]*$`、64文字以内 |
| A5 | SKILL.md が ~100行以内 | 行数カウント (警告: >100, エラー: >150) |
| A6 | 参照ファイルが1階層のみ | references/ 内で他の references/ への参照を検出 |
| A7 | Windows パスがない | `\` パスパターンの検出 |

---

## Phase B: 内容チェック (Claude レビュー)

Claude が手動でレビューする項目。

| # | チェック項目 | 確認観点 |
|---|------------|---------|
| B1 | 一般知識の繰り返しがないか | Claude が既知の情報を冗長に記載していないか |
| B2 | 自由度がタスクの脆弱性に合っているか | 厳密すべき箇所が自由すぎないか、またその逆 |
| B3 | 時間依存の情報が含まれていないか | バージョン番号、日付、URL が陳腐化しないか |
| B4 | 用語が一貫しているか | 同じ概念に異なる用語を使っていないか |
| B5 | ワークフロー主体が明示されているか | コマンド型: 各ステップに 👤/🤖/🗣️ があるか |
| B6 | description が具体的か | トリガー条件と使用タイミングが明確か |
| B7 | 参照リンクが正しいか | SKILL.md のリンクが実際のファイルを指しているか |
| B8 | テンプレートが必要十分か | 過剰な制約や不足がないか |

---

## 検証の実行方法

```bash
# Phase A: 自動検証
bash .claude/skills/skill-creator/scripts/validate-skill-structure.sh .claude/skills/{target-skill}/

# Phase B: Claude レビュー
# 上記チェック項目に基づき、Claude が各ファイルを読んでレビュー
```

---

## 判定基準

| 結果 | 条件 |
|------|------|
| ✅ PASS | Phase A 全件パス + Phase B で重大な問題なし |
| ⚠️ WARNING | Phase A に警告あり、または Phase B で軽微な改善点あり |
| ❌ FAIL | Phase A にエラーあり、または Phase B で重大な問題あり |

# 技術選定リクエスト

**ユーザーの要望:** $ARGUMENTS

---

## Phase 1: 要件ヒアリング

まず、適切な技術選定を行うために以下を確認してください。
不明な点があればユーザーに質問し、要件を明確にしてから調査に進んでください。

### 確認すべき項目

1. **目的・ゴール**
   - 何を実現したいのか？
   - なぜその技術が必要か？

2. **制約条件**
   - 既存システムとの互換性要件はあるか？
   - チームの習熟度・学習コストの許容範囲は？
   - パフォーマンス要件（レイテンシ、スループット等）はあるか？
   - セキュリティ要件はあるか？

3. **運用面**
   - 本番環境か開発環境か？
   - 長期サポート（LTS）が必要か、最新機能が必要か？
   - メンテナンス頻度の許容範囲は？

4. **優先順位**
   - 安定性 vs 最新機能
   - 学習コスト vs 将来性
   - シンプルさ vs 拡張性

### ヒアリング例

```
以下について教えてください：

1. この技術は本番環境で使用しますか？開発環境のみですか？
2. 長期的な安定性と最新機能、どちらを優先しますか？
3. チームメンバーの経験レベルはどの程度ですか？
4. 他に考慮すべき制約はありますか？
```

**重要:** 要件が曖昧な場合は必ず質問してください。推測で選定を進めないでください。

---

## Phase 2: 調査

要件が明確になったら、以下の手順で調査を行います。

1. **WebSearch** で候補技術について検索:
   - 公式サイトのリリース情報
   - endoflife.date でのサポート状況
   - Docker Hub での利用可能なイメージタグ（該当する場合）

2. **WebFetch** で詳細情報を取得:
   - サポート期間・EOL 日
   - LTS / Stable の区分
   - 破壊的変更の有無

3. **現在のプロジェクト設定を確認**（該当ファイルがあれば）:
   - Dockerfile, package.json, Cargo.toml, compose.yaml 等

| 項目 | 値 |
|------|-----|
| **現在の設定** | (使用中のバージョン、なければ「未設定」) |
| **推奨バージョン** | X.Y.Z |
| **最新安定版** | X.Y.Z |
| **サポート終了日** | YYYY-MM-DD |

**根拠 - [ソース名](URL):**

> サイト本文からの引用（原文を転記）

**要件との適合:**
- [要件1] に対して → このバージョンが適切な理由
- [要件2] に対して → このバージョンが適切な理由

**トレードオフ:**
- 採用した場合のメリット
- 採用した場合のデメリット・注意点

**代替案:**

---

## 注意事項

- 必ず公式ソースまたは信頼できる情報源を参照すること
- 「最新だから」ではなく「要件に合っているから」という観点で選定すること
- 根拠が不明確な場合はその旨を明記し、追加調査を提案すること
- 複数の選択肢がある場合は比較表を作成すること
